# I-2.1.9: Spots-remaining badge on `/events/:slug`

**Feature ID:** I-2.1.9
**Module:** 2.1 ‚Äî Event Detail Page
**Status:** ‚úÖ Complete (2026-04-30)
**Dependencies:** I-2.1.1 ‚Äî public event detail page; I-1.2.2 ‚Äî `event_categories.spots_total` / `spots_remaining` columns + CHECK constraints (migration 0016, W1.2); I-2.1.4 ‚Äî pricing breakdown is the render site; I-2.1.7 ‚Äî `useRegistrationState` reused for state-gating
**Phase-3 follow-up:** I-3.2.10 ‚Äî atomic decrement + capacity revalidation. Until that ships, the badge is built but **dark in production** behind a default-`false` env flag (D11). The API contract change (D1) ships regardless ‚Äî additive and forward-compatible.

## Scope

Render two distinct urgency badges per category inside `PublicEventPricingBreakdown` on `/events/:slug` ‚Äî `<Badge variant="destructive">Sold out</Badge>` for `spotsRemaining === 0` and `<Badge variant="outline">{n} spot{s?} remaining</Badge>` for `0 < spotsRemaining ‚â§ threshold`. All web rendering is gated by `VITE_PUBLIC_SPOTS_REMAINING_BADGE_ENABLED` (default `false`); the implementation ships dark and is enabled by a one-line env flip once I-3.2.10 lands.

The public event payload is extended with a wrapped, nullable `capacity: { spotsTotal, spotsRemaining } | null` (forward-compatible with a future organizer "unlimited" toggle), and the API service projects `capacity: null` (with a Pino warn) if a row violates the `remaining ‚â§ total` invariant ‚Äî fail-soft so a corrupted row never 5xxs the entire page.

## Acceptance Criteria

1. **API contract (additive, wrapped, nullable).** `eventPublicCategorySchema` includes `capacity: eventPublicCategoryCapacitySchema.nullable()` where `eventPublicCategoryCapacitySchema = z.object({ spotsTotal: z.number().int().positive(), spotsRemaining: z.number().int().min(0) }).superRefine((v, ctx) => v.spotsRemaining <= v.spotsTotal)`. Today's API always emits a non-null `capacity`; `null` is reserved for the future "unlimited" toggle.
2. **API service projection (fail-soft).** `selectPublicCategories()` (`apps/api/src/modules/events/public-detail-service.ts`) projects `{ spotsTotal, spotsRemaining }` from each DB row. If the row violates `spotsRemaining > spotsTotal` (defensive ‚Äî DB CHECK should make this impossible), the service logs a Pino `warn` with `{ eventId, categoryId, spotsTotal, spotsRemaining }` and projects `capacity: null` for that row only; other rows are unaffected.
3. **Display gate (web).** Per-category badge renders **only** when **all** hold:
   - **Env flag** `publicEnv.VITE_PUBLIC_SPOTS_REMAINING_BADGE_ENABLED === true` (default `false` ‚Üí no badge ever).
   - `category.capacity` is non-null AND
   - `state ‚àà { "open", "not_yet_open" }` (NOT `"unknown"`, NOT `"closed_window"` / `"event_ended"`). Hide on `"unknown"` so cached SSR HTML never advertises a stale availability/sold-out state.
   - One of: `spotsRemaining === 0` ‚Üí sold-out; `0 < spotsRemaining ‚â§ threshold` ‚Üí low-spots; otherwise ‚Üí no badge.
4. **Label format.** Exactly `"X spots remaining"`. Singular-aware: `"1 spot remaining"` when count is exactly 1.
5. **Per-row placement.**
   - **Desktop table:** an additional `Availability` column appears whenever **at least one** row has a visible badge (mirrors the I-2.1.4 `hasEarlyBird` conditional column pattern). When no row qualifies, the column is omitted.
   - **Mobile card:** an additional `<dt>Availability</dt>/<dd>{badge}</dd>` is appended to the row's `<dl>` whenever that row's badge is visible (per-row conditional, no header).
6. **Visual + A11y.**
   - Sold-out: `<Badge variant="destructive">` + `lucide-react` `Ban` icon (`aria-hidden`); visible text `"Sold out"`.
   - Low-spots: `<Badge variant="outline">` with `border-amber-500/40 text-amber-700 dark:text-amber-400` urgency tint + `lucide-react` `Users` icon (`aria-hidden`); visible text `"{n} spot{s?} remaining"`.
   - **No `aria-live`** ‚Äî the badge mutates only on page revalidation, not via a client-side timer.
   - `data-testid="spots-remaining-badge"`, `data-category-slug={slug}`, **`data-surface="desktop" | "mobile"`** to disambiguate the dual-rendered surfaces.
7. **No new client-side polling, no new env reads beyond the new flag, no per-user response.** Pure transform of the loader payload + `useRegistrationState`. The `useRegistrationState` chain inside `PublicEventPricingBreakdown` is the **third** concurrent chain on the page (sidebar CTA + mobile CTA + breakdown) ‚Äî explicit and intentional.
8. **CDN cache contract preserved.** The badge is a pure transform of the SSR loader payload; no `Vary` headers, no `s-maxage` reduction. Counts refresh on the next CDN revalidation (60s `s-maxage` + 300s `stale-while-revalidate` from I-2.1.1). Layout-shift consequence of the SSR-`"unknown"` gate (column appears post-hydration) accepted; mirrors I-2.1.4's `Active`/`Expired` post-mount appearance.
9. **Determinism + idempotency.** Given the same `(category.capacity, state, env-flag)` the rendered DOM is byte-identical across renders.
10. No edits outside `packages/shared/**`, `apps/api/**`, `apps/web/**`, the new env-flag declaration, and the three docs files.
11. `pnpm --filter @repo/shared check-types|test`, `pnpm --filter api check-types|test`, `pnpm --filter web check-types|lint|test` all pass with no regressions.

## Key Decisions

- **D1 ‚Äî Wrapped, nullable `capacity` on the public schema.** Two flat non-nullable fields (`spotsTotal`, `spotsRemaining`) was rejected post plan-review (Important-1) ‚Äî once required-number clients exist, switching to nullable later for the future "unlimited" toggle would be a breaking change. Wrapping as `capacity: { spotsTotal, spotsRemaining } | null` makes the future-state toggle a pure additive change to the projection conditional; atomic also closes the impossible-state where one of the two fields would be null and the other not.
- **D2 ‚Äî Display threshold is `min(25, ceil(0.20 √ó spotsTotal))` (absolute cap + 20% ratio).** Hybrid: `total=10` ‚Üí 2; `total=100` ‚Üí 20; `total=125` ‚Üí 25 (cap engages); `total=1000` ‚Üí 25. Helper-level constants (`SPOTS_REMAINING_MAX_DISPLAY_THRESHOLD = 25`, `SPOTS_REMAINING_DISPLAY_THRESHOLD_RATIO = 0.20`) so future product tweaks are a one-file change. Edge handling (per Important-6): `total <= 0` returns `0` (fail-safe ‚Üí never display); `total = 1` returns `1`. The threshold is a literal cliff (`<=` boundary): `remaining = threshold` shows; `remaining = threshold + 1` hides.
- **D3 ‚Äî Hide on `"unknown"` SSR baseline AND on `"closed_window"` / `"event_ended"`.** Adopted post plan-review (Critical-1). The count itself is static, but it's meaningful only inside an active registration window ‚Äî and cached HTML can survive past a window-close boundary, at which point a "3 spots remaining" badge would actively mislead a screen reader / cached-page reader. Mirrors I-2.1.7's SSR-`"unknown"` discipline. Layout-shift cost (column appears post-hydration) accepted ‚Äî no worse than I-2.1.4's `Active`/`Expired` post-mount appearance.
- **D4 ‚Äî Render `<Badge variant="destructive">Sold out</Badge>` for `spotsRemaining === 0` (in displayable states).** Adopted post plan-review (Important-3). Silently hiding a zero-remaining row would leave an active-looking price with no signal that the user can't act on it. Per-category sold-out is a *capacity* concern, naturally owned here ‚Äî distinct from I-2.2.3's *event-level* status enum (which describes the entire event being sold-out / cancelled / postponed, not per-category). When I-2.2.3 ships, the two surfaces coexist at different scopes.
- **D5 ‚Äî Render server-side, gated only by registration state (no `useNow()`).** Capacity counts are static integers from the loader payload ‚Äî they don't depend on `now`. The only non-static input is `state` (which transitions at registration-window boundaries). Server-rendering is therefore safe **once** D3's `"unknown"` gate is in place: the badge appears post-mount when `state` resolves to `open` / `not_yet_open` and stays put until a boundary fires. No hydration mismatch risk.
- **D6 ‚Äî Reuse `useRegistrationState` inside `PublicEventPricingBreakdown` (third concurrent timer chain ‚Äî explicit, intentional).** Lighter-touch alternative to lifting state to the page (which would touch `PublicEventPage`, `PublicEventRegisterCta`, `PublicEventStickyMobileCta`, and their tests ‚Äî explicitly out of scope). Boundaries fire at most twice per event lifecycle; three concurrent chains all firing at the same moment is negligible. A future cleanup PR can lift state to the page when there's a natural reason to refactor the CTA tree.
- **D7 ‚Äî Conditional desktop column header (`hasAnySpotsBadge`) + per-row inline mobile.** Mirrors the I-2.1.4 `hasEarlyBird` pattern exactly so the breakdown's responsive shape stays consistent. Three-row events where one row qualifies ‚Üí desktop has an `Availability` column with two `‚Äî` cells + one badge cell; mobile drops the row's `<dt>/<dd>` for non-qualifying rows.
- **D8 ‚Äî Singular-form pluralization in the helper, not the component.** `formatSpotsRemainingLabel(count)` returns `"1 spot remaining"` vs `"3 spots remaining"`. English-only V1 (matches `eventTypeSchema` etc.); no `Intl.PluralRules`.
- **D9 ‚Äî Schema stays strict (`superRefine(remaining <= total)`); API service projects `capacity: null` defensively if the row violates the invariant.** Adopted post plan-review (Important-2). The schema cannot accept `remaining > total` because the resulting badge would compute a negative or impossible threshold; defensive projection in the service ensures an upstream bug never 5xxs the page. Logged via Pino `warn` so ops can investigate. Regression test covers both happy-path and fail-soft path.
- **D10 ‚Äî `Users` icon (low-spots) + `Ban` icon (sold-out).** Both lucide-react, already in app; no new dependency. Distinct icons reinforce the distinct semantics.
- **D11 ‚Äî Env-flag gate `VITE_PUBLIC_SPOTS_REMAINING_BADGE_ENABLED` (default `false`).** Adopted post plan-review (Critical-2). The v1-implementation-plan row literally says "build behind feature flag if Phase 3 not ready" ‚Äî and Phase 3 (I-3.2.10 atomic decrement + capacity revalidation) is not ready. The flag lives in `apps/web/src/lib/env/public.ts` so it's tree-shakable and missing-flag means feature-off. Production deploys the implementation but the badge is invisible until the flag flips. **Implementation note:** The flag uses a custom `stringBoolEnv` parser (NOT `z.coerce.boolean()`) ‚Äî Vite inlines env values as strings, and `z.coerce.boolean()` uses JS truthiness so `"false"` / `"0"` would parse to `true` and silently enable the badge. `stringBoolEnv` accepts only `"true"` / `"1"` (case-insensitive) as `true`; everything else (including missing, empty, `"false"`, `"0"`, unknown strings) is `false`. This was the Critical finding adopted from post-impl GPT-5.5 code review.
