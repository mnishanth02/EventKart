# I-2.1.10: Early-bird countdown timer on `/events/:slug`

**Feature ID:** I-2.1.10
**Module:** 2.1 ‚Äî Event Detail Page
**Status:** ‚úÖ Complete (2026-04-30)
**Dependencies:** I-2.1.1 ‚Äî public event detail page; I-1.2.3 ‚Äî pricing tiers w/ early-bird; I-2.1.4 ‚Äî `pricing.ts` `hasValidEarlyBirdOffer` legacy guard reused; I-2.1.7 ‚Äî sidebar + mobile CTA chassis hosting the badge, chained-`setTimeout` boundary refresh pattern reused

## Scope

Render an SSR-safe, real-data-only countdown badge ("Early-bird closes in 2d 3h") inside the public event CTA ‚Äî both the desktop sidebar Card and the mobile sticky bottom bar ‚Äî whenever the event has at least one early-bird tier whose effective cutoff is still in the future. Refreshes once per minute (top-of-minute aligned, no drift) and re-derives from the deadline; auto-rolls to the next eligible tier when the soonest cutoff passes; auto-hides when no eligible tier remains. Web-only; no schema/API/package changes.

## Acceptance Criteria

1. **Real data only** ‚Äî never render the countdown when:
   - No pricing tiers, OR
   - No tier has a valid early-bird offer (`hasValidEarlyBirdOffer` ‚Üí `false` for all tiers ‚Äî this re-uses the I-2.1.4 legacy-guard predicate so a tier with `earlyBirdPrice >= basePrice`, a null deadline, or a NaN deadline is filtered out and the badge cannot disagree with the breakdown column), OR
   - All eligible tiers' **effective cutoffs** (see AC 2) are `<= now`, OR
   - Registration state is `closed_window` or `event_ended` (registration unavailable ‚Üí countdown is moot ‚Äî defence in depth on top of AC 2's effective-cutoff math, mirroring I-2.1.7 D7 price-hint gating), OR
   - The component is in SSR / pre-mount baseline (cached HTML must never advertise a stale countdown).
2. **Effective cutoff per tier** (the badge must never over-promise time the user cannot actually use):
   - `effectiveCutoffMs = min(parse(earlyBirdDeadline), parse(registrationClosesAt) ?? +Infinity, parse(endAt))`.
   - A tier is **eligible** when (a) `hasValidEarlyBirdOffer(tier) === true`, (b) `effectiveCutoffMs > now`, AND (c) `registrationOpensAt === null || parse(registrationOpensAt) < effectiveCutoffMs` (offer must be reachable from inside the registration window).
   - Among eligible tiers, pick the **soonest `effectiveCutoffMs`**. On exact tie, keep insertion order (deterministic fallback). The label counts down to that cutoff, not to the raw `earlyBirdDeadline`.
3. **Format** (matches spec example "Early-bird closes in 2d 3h"):
   - `‚â• 24h`: `"Xd Yh"` (always emit both units; "2d 0h" is acceptable to keep the format stable across boundaries).
   - `‚â• 1h, < 24h`: `"Xh Ym"` (always emit both units).
   - `‚â• 1m, < 1h`: `"Xm"`.
   - `< 1m, > 0`: `"<1m"`.
   - When `now >= effectiveCutoffMs` the helper returns `null` and the component hides for that tier (the next eligible tier ‚Äî if any ‚Äî takes over on the next tick).
4. **SSR-safe** ‚Äî `useEarlyBirdCountdown` returns `null` on SSR + first render. Cached HTML must not contain "Early-bird closes" text. Verified by `renderToString`.
5. **Tick scheduling** ‚Äî single chained `setTimeout` that fires precisely at each label transition (no drift):
   - `currentFloorMin = floor(msUntilCutoff / 60_000)`; `delay = msUntilCutoff ‚àí currentFloorMin*60_000 + 1` (the `+1ms` lands one ms past the bucket boundary so the floor recomputation registers as the lower bucket).
   - This formula is **always `‚â§ 60_001 ms`** by construction (the residue is `< 60_000`), so a `MAX_BOUNDARY_DELAY_MS` cap is not needed. Defence-in-depth: still floor at `Math.max(1, ‚Ä¶)` to avoid a `0` delay if a future change accidentally makes the residue zero.
   - Cleanup on unmount and on effect dep change. No `setInterval` (drift + double-render risk).
   - **No global per-minute `+1ms`** ‚Äî that would undercount the visible minute by one in `‚â•1h` buckets (e.g. `2h 1m` instead of `2h 2m`). The `+1ms` exists *inside* the residue calculation to cross the bucket exactly.
6. **Placement** ‚Äî single `PublicEventEarlyBirdCountdown({ event, state })` component rendered inside both `PublicEventRegisterCta` (sidebar) and `PublicEventStickyMobileCta` (mobile bar) **between the subtitle and the price hint**. The two surfaces share the component (single source of truth, can never drift).
7. **Visual + A11y** ‚Äî compact `<Badge variant="secondary">` with a decorative `lucide-react` `Clock` icon (`aria-hidden`) and visible text `"Early-bird closes in 2d 3h"`. **No `aria-live`** ‚Äî both surfaces render in the DOM (sidebar `<aside>` + mobile fixed bar with `lg:hidden`), so a polite live region on each would double-announce. Visible text is sufficient; screen readers reading the CTA card pick it up on focus, and the absolute deadline timestamp is already in the breakdown card (I-2.1.4). The compact `"2d 3h"` is read by SR engines as "two D three H" ‚Äî acceptable trade-off for visual compactness.
8. **No env reads, no I/O, no random IDs** ‚Äî pure function of `(tiers, event, now)`. Preserves the I-2.1.1 CDN cache contract (no extra `Vary`, no shorter `s-maxage`, no per-user response).
9. **Determinism + idempotency** ‚Äî re-renders with the same `(tiers, event, label)` produce identical DOM.
10. No edits outside `apps/web/**` and the three docs files (`progress.md`, `docs/v1-implementation-plan.md`, this impl-plan file).
11. `pnpm --filter web check-types`, `lint`, `test` all pass with no regressions.

## Key Decisions

- **D1 ‚Äî Single source of truth helper (`early-bird.ts`).** `getNextActiveEarlyBirdCutoff(tiers, event, now)` returns `{ cutoffMs, categorySlug }` for the soonest **eligible** tier (or `null`). Reuses `hasValidEarlyBirdOffer` from `pricing.ts` so a tier with `earlyBirdPrice >= basePrice`, a null deadline, or a NaN deadline is filtered out ‚Äî matches the I-2.1.4 breakdown badge contract. Two surfaces (sidebar + mobile) share one component which calls one hook which calls one helper ‚Äî no opportunity for divergent display.
- **D2 ‚Äî Cutoff = `min(deadline, registrationClosesAt, endAt)`.** The badge must not promise time the user cannot use. If `registrationClosesAt` is earlier than `earlyBirdDeadline`, the offer becomes unusable at close, not at the raw deadline. Same for `endAt`. A tier whose `registrationOpensAt >= effectiveCutoff` is filtered out entirely (offer is never reachable). This is the **Critical-1** fix from the plan review ‚Äî without it the countdown overstates urgency on misconfigured events.
- **D3 ‚Äî Hide in `closed_window` / `event_ended`.** Defence in depth on top of D2's effective-cutoff math: even if a tier's effective cutoff is somehow still in the future when registration is already closed, the countdown is moot. Mirrors I-2.1.7 D7 which hides the price hint in the same states.
- **D4 ‚Äî SSR baseline returns `null` (no element in cached HTML).** Mirrors `useNow()` (I-2.1.4) and `useRegistrationState()` (I-2.1.7). A cached "Early-bird closes in 5h 23m" five days later would be a lie. The cost is the badge appears post-mount with a brief layout shift ‚Äî accepted; the badge is small and the CTA card height already varies with state.
- **D5 ‚Äî Tick formula `delay = msUntilCutoff ‚àí currentFloorMin*60_000 + 1`.** Fires precisely at each `floor((cutoff ‚àí now) / 60_000)` transition with **no global per-minute `+1ms` drift** (the **Critical-2** fix from the plan review ‚Äî a global `+1ms` would undercount displayed minutes by one in `‚â•1h` buckets). The residue is bounded by `60_000`, so `delay ‚â§ 60_001 ms` by construction; no separate `MAX_BOUNDARY_DELAY_MS` cap needed. Floor at `Math.max(1, ‚Ä¶)` for defence in depth.
- **D6 ‚Äî No `aria-live` on the dual-surface badge.** The component renders on **both** the sidebar `<aside>` and the mobile fixed bottom bar (gated on different breakpoints visually but **both present in the DOM**). A polite live region on each would double-announce every minute. The visible text is sufficient ‚Äî screen readers reading the CTA card pick it up on focus, and the absolute deadline timestamp is already announced inside the breakdown card from I-2.1.4. Decorative `Clock` icon is `aria-hidden`. This is the **Important-3** fix from the plan review.
- **D7 ‚Äî Multi-tier: soonest effective cutoff wins.** If two tiers have different cutoffs, the user's effective urgency is whichever closes first ‚Äî the cheapest offer might still be available later if the other tier's cutoff triggers first. Picking the soonest is conservative + matches the user's mental model of "what's the next thing I'd lose". On exact tie, insertion order is the deterministic fallback.
- **D8 ‚Äî No new dependencies.** `lucide-react` is already used across the app (header, sidebar, error chrome, admin, organizer); reusing the `Clock` icon keeps the badge consistent with the rest of the codebase.

## Known Follow-Ups

- The `<1m` bucket is intentionally coarse ‚Äî V1 is not building a seconds-precision live counter (out of scope; the breakdown card already shows the absolute deadline timestamp via I-2.1.4 `Until ‚Ä¶`).
- The `closed_window` / `event_ended` gate (D3) hides the countdown even if a misconfigured event has an early-bird deadline later than `registrationClosesAt`. Same posture as I-2.1.7 D7 (price-hint gating); we surface no UI for the misconfiguration.
- Same-millisecond ties between two tiers' effective cutoffs fall back to insertion order ‚Äî deterministic but not user-meaningful. Acceptable; in practice deadlines collide at organizer-chosen wall-clock minutes, not milliseconds.
- Module 2.1 has **only I-2.1.9 (spots-remaining badge)** remaining after this slice ‚Äî that one needs I-1.2.2 capacity columns (already shipped via W1.2 of the Phase-2 Readiness sprint) plus I-3.2.10 atomic decrement (Phase 3) before it can be wired truthfully.
